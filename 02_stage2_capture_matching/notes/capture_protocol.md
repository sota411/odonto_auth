# 実写口腔内画像 撮影・管理プロトコル

- 文書版: 1.0
- 制定日: 2026-07-11
- 適用範囲: 同意を得た18歳以上の協力者から収集する、歯牙補助認証研究用の実写口腔内画像

## 目的と実施条件

本手順は、スマートフォンで撮影した前歯画像について、同じ人物を日を分けて照合できるデータを収集するためのものである。1人につき5セッションを別の日に実施し、各セッションで受入基準を満たす画像を10枚残す。合計は1人50枚である。

公開データセットだけでは、補助認証用途への同意、複数日の撮影、前歯の歯牙単位マスクを同時に満たせない。そのため、本手順で収集するデータは公開データと混在させない。COdeなどの公開データを使う評価にも、本手順の被験者IDを流用しない。

実写画像、歯牙の切り出し画像、注釈、metadataは生体情報として扱い、公開リポジトリ、クラウドストレージ、チャット、メールには置かない。公開するのは、再現に必要なコード、個人を識別できない集計値、公開前に別途確認した学習済みモデルに限る。

## 実施体制

| 役割 | 作業 |
|---|---|
| 研究責任者 | 同意書の版管理、匿名IDの発行、対応表の管理、撤回・削除の実行、本収集の開始承認 |
| 撮影担当者 | 同意の確認、機器設定、撮影、一次確認、端末からの削除 |
| 確認担当者 | metadataと画像の照合、顔写り込み・画質・セッション分離の確認 |
| 協力者 | 同意後の撮影参加。撮影中でも理由を示さず中止できる |

同じ人が複数の担当を兼ねてもよい。ただし、撮影担当者が入力したmetadataは、原則として別の確認担当者が確認する。担当者名は画像metadataへ書かず、`op-`から始まる担当者IDを使う。

## 匿名IDと保管場所

研究責任者は、氏名と無関係な8桁の小文字16進数を発行し、`sub-1a2b3c4d`の形式で`subject_id`とする。連番、イニシャル、誕生日、学籍番号は使わない。セッションIDと画像IDは次の規則で決める。

```text
subject_id:  sub-1a2b3c4d
session_id:  sub-1a2b3c4d_s01
image_id:    sub-1a2b3c4d_s01_i01
```

氏名、連絡先、署名、`subject_id`の対応表は、画像データとは別の暗号化されたローカル媒体に保存する。対応表をGit管理下やdatasetのディレクトリへ置かない。撮影担当者へ渡す情報は`subject_id`、対象セッション番号、同意書版だけにする。

ローカルデータの基準ディレクトリは次のとおりとする。`01_stage1_real_image_extraction/datasets/`全体は既存の`.gitignore`で除外されている。

```text
01_stage1_real_image_extraction/datasets/dataset_real/
├── raw/
│   └── <subject_id>/
│       ├── enrollment/<session_id>/<image_id>.jpg
│       └── verification/<session_id>/<image_id>.jpg
├── metadata.csv
└── annotations/
    └── yolo/<image_id>.txt
```

`annotations/yolo/`は後工程で使う。画像と同じstemのテキストファイルを対応させ、クラス名は既存実装の`R1`、`R2`、`R3`、`L1`、`L2`、`L3`に合わせる。実写画像を擬似画像の`train`または`test`へ直接コピーしない。

## セッション割当

日付はセッションごとに変える。同じ暦日の撮り直しは同じセッションとして扱い、翌日以降に進むまで次のセッション番号を発行しない。時刻が24時間以上空いていても同じ暦日なら別セッションとは数えない。

| セッション | 用途 | 日付条件 | 推奨する照明 | 推奨する撮影タイミング |
|---|---|---|---|---|
| `s01` | `enrollment` | 1日目 | `indoor` | `after_brushing` |
| `s02` | `enrollment` | s01の翌日以降 | `natural` | `after_meal` |
| `s03` | `verification` | s02の翌日以降 | `indoor` | `after_waking` |
| `s04` | `verification` | s03の翌日以降 | `natural` | `after_brushing` |
| `s05` | `verification` | s04の翌日以降 | `flash` | `after_meal` |

照明と撮影タイミングは、本人内変動を測るためにセッション間で変える。安全上または生活上の理由で推奨条件を使えないときは、無理に合わせず、実際の値をmetadataへ記録する。フラッシュで痛みや不快感が出る場合は直ちに中止し、`indoor`へ切り替える。

`s01`と`s02`の画像だけで登録テンプレートを作り、`s03`から`s05`だけを検証クエリに使う。用途はセッション単位で固定する。同じセッションの一部を`enrollment`、残りを`verification`に分けてはならない。

## 撮影前チェックリスト

撮影担当者は、各項目を上から確認する。

- [ ] 協力者は18歳以上であり、本日も参加を希望している。
- [ ] 所属機関の倫理審査要否を確認し、承認番号または審査不要・免除判断の記録が有効である。
- [ ] 署名済み同意書の版が1.0以降で、撤回連絡先と保存期限が記入されている。
- [ ] 同意書の対応表で`subject_id`を確認した。氏名を撮影用フォルダやmetadataへ入力していない。
- [ ] 本日のセッション番号は前セッションより大きく、前セッションとは日付が異なる。
- [ ] `s01`、`s02`は`enrollment`、`s03`から`s05`は`verification`としてフォルダを作った。
- [ ] 端末の写真クラウド同期、位置情報付与、Live Photoまたはモーションフォト、美肌補正、フィルターを無効にした。
- [ ] レンズを清掃し、横持ちに固定した。デジタルズームは1.0倍である。
- [ ] 撮影場所に第三者の顔、名札、郵便物、画面、鏡などの識別情報が写らない。
- [ ] 清潔な定規で15cmから20cmの距離を確認した後、定規を画角から外した。
- [ ] 協力者へ、痛みや疲れを感じたらすぐに止められると再度伝えた。

急な歯痛、出血、口内炎の悪化、顎の痛み、撮影による強い不快感がある場合は、その日の撮影を開始しない。撮影は診療行為ではないため、担当者は診断を行わない。

## 撮影手順

1. 協力者は座った姿勢を取り、カメラを横向きにする。撮影担当者が保持しても、協力者が自分で保持してもよい。
2. カメラと前歯の距離を15cmから20cmにする。端末は歯列正面に置き、上下左右の傾きをできるだけ小さくする。
3. 協力者は「イー」と発音するときの口形を作る。器具で唇や頬を引かない。前歯6本が中央に入り、目、鼻全体、耳が入らない構図にする。
4. 1枚撮影するたびに画面を確認する。受入基準を満たさない画像はその場で削除し、同じ画像番号で撮り直す。
5. 受入画像を10枚得るまで繰り返す。各画像では数cm以内の平行移動や小さな角度差を許すが、距離、口形、照明条件は同一セッション内で変えない。
6. 10枚を`i01`から`i10`へ改名し、所定のフォルダへ転送する。元の撮影日時や氏名をファイル名に含めない。
7. 1画像1行でmetadataを記入する。推測値や前回値のコピーで空欄を埋めない。

## 画像の受入基準

次の全項目を満たす画像だけを残す。

- 横向きで、前歯6本の歯冠が中央付近に写っている。解剖学的に存在しない歯や治療による欠損がある場合は、見える範囲を撮影し、診断名を書かず`notes`に`visible_teeth_limited`と記録する。
- 歯の輪郭を確認でき、手ぶれ、ピンぼけ、白飛び、暗部つぶれがない。
- 目、鼻全体、耳など、顔認識に使える範囲が写っていない。唇と口周辺の最小範囲は許容する。
- 第三者、鏡像、通知、位置情報、氏名などの識別情報が写っていない。
- 画像編集、色補正、フィルター、デジタルズームを使っていない。
- JPEGとして開ける。向きが正しく、EXIF、GPS、サムネイルなどの付随情報を削除済みである。

撮影直後の端末上ではEXIF削除前でもよい。ローカル保管先へ入れる前に付随情報を削除し、表示方向が変わっていないことを確認する。

## 顔が写り込んだ場合

顔が受入範囲を超えて写った画像は、次の順で処理する。

1. その場で再撮影できる場合は、該当画像を端末と「最近削除した項目」から削除し、撮り直す。採用画像の`face_action`は`retaken`とする。
2. 再撮影できず研究責任者が画像を残す必要を認めた場合に限り、ネットワークへ接続しない端末で口腔領域だけに切り詰める。残る顔領域は不透明な単色で完全に塗りつぶす。ぼかし、モザイク、半透明マスクは使わない。
3. 確認担当者は、目、鼻全体、耳、輪郭、鏡像、EXIFサムネイルが残っていないことを確認する。採用画像の`face_action`は`masked`とする。
4. 確認後、加工前画像を端末、作業フォルダ、画像編集ソフトの履歴、「最近削除した項目」から削除する。加工済み画像だけを所定の`raw/`へ移す。

再撮影も不可逆マスキングも完了できない画像は採用しない。

## 転送と保管

1. USBケーブルまたは暗号化されたローカル媒体で、研究責任者が管理する暗号化ディスクへ転送する。一般向けクラウド写真同期、共有アルバム、メール、チャットは使わない。
2. 転送後に画像を1枚ずつ開き、枚数、向き、ファイル名を確認する。SHA-256を計算し、metadataの`sha256`へ記録する。
3. metadataの検証が通り、暗号化されたローカルバックアップを1部作成した後、撮影端末と「最近削除した項目」から画像を削除する。
4. 画像、metadata、注釈、歯牙切り出しを保存できるのは、研究責任者が承認した担当者だけとする。アクセス権を半年ごとと担当変更時に見直す。
5. 現行計画の研究終了日は2027-03-12である。実写画像、metadata、注釈、切り出し画像は研究終了後3年間保存し、2030-03-31までに主媒体、バックアップ、作業コピーを削除する。計画終了日を変更する場合は、同意書の版を更新し、既存協力者から保存期限変更への同意を得る。

削除時は、対象`subject_id`、削除日、削除担当者、削除対象、確認担当者だけを削除記録に残す。氏名や画像のハッシュは削除記録に残さない。

## metadata定義

trackedな空テンプレートは`02_stage2_capture_matching/fixtures/capture_metadata_template.csv`に置く。実値を記入した`metadata.csv`はdataset側に置き、Gitへ追加しない。

| 列 | 必須・形式 | 内容 |
|---|---|---|
| `schema_version` | 必須、`1` | この表の版 |
| `collection_phase` | 必須、`pilot`または`main` | パイロットと本収集の混入防止 |
| `subject_id` | 必須、`sub-[0-9a-f]{8}` | 匿名被験者ID |
| `session_id` | 必須、`<subject_id>_s01`から`s05` | 被験者内セッションID |
| `session_index` | 必須、整数1から5 | 撮影順 |
| `capture_role` | 必須、`enrollment`または`verification` | 登録用か検証用か |
| `capture_date` | 必須、`YYYY-MM-DD` | 撮影日。時刻と位置は記録しない |
| `image_id` | 必須、`<session_id>_i01`から`i10` | 画像ID |
| `image_index` | 必須、整数1から10 | セッション内の採用順 |
| `relative_path` | 必須、相対パス | `raw/<subject_id>/<capture_role>/<session_id>/<image_id>.jpg` |
| `sha256` | 必須、小文字16進64桁 | 保管後の画像内容のハッシュ |
| `device_model` | 必須、自由記述 | 端末の製品名だけ。所有者名、シリアル、電話番号を書かない |
| `camera_app` | 必須、`system_camera` | 標準カメラアプリを使う |
| `lighting` | 必須、`indoor`、`natural`、`flash`のいずれか | 実際に使った主照明 |
| `capture_timing` | 必須、`after_waking`、`after_meal`、`after_brushing`、`other` | 撮影直前の状態に最も近いもの |
| `distance_cm` | 必須、整数15から20 | 測った撮影距離 |
| `orientation` | 必須、`landscape` | 端末の向き |
| `mouth_pose` | 必須、`ee` | 「イー」の口形 |
| `saliva_level` | 必須、`normal`または`high` | 画像上で確認した状態 |
| `food_debris` | 必須、`absent`または`present` | 画像上で確認した状態 |
| `face_action` | 必須、`not_needed`、`retaken`、`masked` | 顔写り込みへの処理 |
| `quality_status` | 必須、`accepted` | 受入画像だけをmetadataへ載せる |
| `consent_version` | 必須、署名済み版番号 | この撮影に適用した同意書版 |
| `captured_by` | 必須、`op-`で始まるID | 撮影・入力担当者 |
| `reviewed_by` | 必須、`op-`で始まるID | 画像とmetadataの確認担当者 |
| `notes` | 任意 | 氏名、診断名、連絡先を書かない。複数タグは`;`で区切る。推奨条件を変えたときは`condition_exception:<理由>`を記録する |

`notes`以外は空欄を許可しない。複数の値を推測で補完せず、確認できない画像は採用対象から外す。

## metadataと画像の検証

各セッションを保管先へ取り込む前に、確認担当者が共通検証を行う。本収集の`s05`終了後には、被験者単位の完了検証も行う。1件でも不一致があれば取り込みまたは評価利用を止め、画像またはmetadataを修正して最初から確認する。

### 全フェーズ共通

1. CSVをUTF-8、ヘッダ1行、1画像1行として標準CSVパーサで読み込めることを確認する。表計算ソフト上の見た目だけで判定しない。
2. ヘッダがテンプレートと完全一致し、`notes`以外に空欄がないことを確認する。
3. `subject_id`、`session_id`、`image_id`、各列の列挙値、日付、数値、SHA-256がmetadata定義の形式に一致することを確認する。
4. `image_id`、`relative_path`、`sha256`がそれぞれCSV内で重複していないことを確認する。
5. 各セッションが10行ちょうどで、`image_index`が1から10まで重複なく存在することを確認する。
6. 各セッション内で`schema_version`、`collection_phase`、`subject_id`、`session_id`、`session_index`、`capture_role`、`capture_date`、`device_model`、`lighting`、`capture_timing`、`distance_cm`、`orientation`、`mouth_pose`、`consent_version`が1種類だけであることを確認する。
7. 存在するセッションについて、`s01`と`s02`は全行`enrollment`、`s03`から`s05`は全行`verification`であることを確認する。同じ`session_id`が両方の用途に出た場合は失敗とする。
8. `relative_path`が基準ディレクトリ内のJPEGを指し、シンボリックリンクではないこと、拡張子と実際の形式がJPEGであることを確認する。
9. 各画像からSHA-256を再計算し、`sha256`と一致することを確認する。ファイル数がCSV行数と同じで、CSVにない余分な画像がないことも確認する。
10. 確認担当者が全画像を目視し、受入基準、顔写り込み処理、EXIF削除を確認する。`face_action=masked`の画像は加工前画像が残っていないことも確認する。
11. `collection_phase=pilot`と`main`を同じ評価入力へ入れない。パイロット画像を本評価へ採用する場合は、本収集開始承認後に同意範囲と品質を再確認し、`main`として撮り直す。

### 本収集完了時

12. 各`subject_id`に`s01`から`s05`が1つずつあり、撮影日が5つの異なる日付で昇順になっていることを確認する。
13. 各`subject_id`が50行ちょうどで、`enrollment`が20行、`verification`が30行であることを確認する。
14. 5セッションを通して`lighting`と`capture_timing`がそれぞれ2種類以上あることを確認する。満たさない場合は再撮影する。安全上の理由で再撮影しない場合は、研究責任者が全10行の`notes`に同じ`condition_exception:<理由>`を記録して承認する。

登録・検証分離はディレクトリ、`capture_role`、セッション番号の3点で確認する。後段の照合評価でも、genuine pairの`query_session_id`と`template_session_id`が同じ行は除外する。

## 撤回対応

協力者から撤回連絡を受けた研究責任者は、同意書の対応表で`subject_id`を特定し、30日以内に次を行う。

1. 新しい撮影、解析、学習、評価への利用を直ちに停止する。
2. 主媒体、バックアップ、作業フォルダから、該当`subject_id`の画像、metadata行、注釈、切り出し画像を削除する。
3. 該当データを含む未公開の派生データとモデルを廃棄するか、該当データを除いて作り直す。
4. 削除記録を作り、完了日を協力者へ連絡する。

すでに公表した集計値、論文、配布済みモデルを回収できない場合がある。この制限は同意取得時に説明する。ただし、撤回後の新しい成果物には対象データを使わない。
